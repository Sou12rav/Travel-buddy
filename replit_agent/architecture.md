# Architecture Documentation

## Overview

Buddy is an AI-powered travel assistant application focused on the Indian travel market. The application provides users with location-based recommendations, travel itineraries, real-time weather updates, and conversational AI assistance. It follows a client-server architecture with a React frontend and Express.js backend.

The application is designed to be responsive and mobile-friendly, with a focus on providing a seamless travel planning experience through an intuitive chat interface augmented with AI capabilities.

## System Architecture

### High-level Architecture

The application follows a modern full-stack architecture with the following components:

- **Frontend**: React application with TypeScript and Tailwind CSS
- **Backend**: Express.js server built with TypeScript
- **Database**: PostgreSQL with Drizzle ORM for schema management
- **External Services**: OpenAI API for conversation capabilities
- **Build System**: Vite for frontend bundling and ESBuild for backend bundling

## Key Components

### Frontend Components

1. **UI Framework**
   - The application uses [shadcn/ui](https://ui.shadcn.com/) components built on Radix UI primitives
   - Components are styled with Tailwind CSS using a customized theme

2. **State Management**
   - React context API is used for global state management via `AppProvider`
   - TanStack Query (React Query) for server state management and data fetching

3. **Routing**
   - Uses the Wouter library for client-side routing with hooks like `useRoute` and `useLocation`

4. **Key Pages**
   - `Home`: Main dashboard with location information, weather, and travel suggestions
   - `Chat`: Conversational interface to interact with the AI assistant
   - `Itinerary`: View and manage travel plans
   - `Profile`: User settings and preferences

### Backend Components

1. **Server Framework**
   - Express.js HTTP server with JSON middleware
   - Request logging and error handling middleware

2. **API Routes**
   - RESTful API design pattern
   - Routes for users, conversations, messages, itineraries, and external data like weather

3. **Storage Layer**
   - Interface-based design with `IStorage` defining data operations
   - Implementation using PostgreSQL via Drizzle ORM
   - In-memory fallback implementation for development

4. **AI Integration**
   - OpenAI client for generating conversational responses
   - Structured prompt engineering for travel-specific assistance

### Database Schema

The application uses a PostgreSQL database with the following key tables defined using Drizzle ORM:

1. **users**
   - Primary user account information
   - Stores authentication details and preferences

2. **conversations**
   - Represents chat sessions between users and the AI assistant
   - Linked to user accounts

3. **messages**
   - Individual messages within conversations
   - Contains content, role (user/assistant), and timestamps

4. **itineraries**
   - Travel plans created for users
   - Contains destination and timing information

5. **savedPlaces**
   - Locations bookmarked by users for future reference

## Data Flow

### Authentication Flow

1. User registers or logs in via the Profile page
2. Credentials are validated against the users table
3. Session is maintained using client-side state and cookies
4. Protected routes check for authenticated state

### Chat Conversation Flow

1. User initiates a conversation from the Home or Chat page
2. A new conversation record is created in the database
3. User messages are sent to the backend API
4. Backend processes the message and sends it to OpenAI API
5. Response is stored in the database and returned to the client
6. UI updates to display the new message with typing indicators

### Itinerary Creation Flow

1. User requests a travel plan through the chat interface
2. AI assistant generates a structured itinerary plan
3. Plan is stored in the itineraries table
4. User can view and modify the itinerary in the dedicated page
5. Changes are synchronized back to the database

## Deployment Strategy

The application is configured for deployment on the Replit platform with the following setup:

1. **Development Mode**
   - Uses Vite's development server with hot module replacement
   - Express backend proxies API requests

2. **Production Build**
   - Frontend: Vite bundles the React application to static assets
   - Backend: ESBuild creates a single bundle for the server

3. **Deployment Target**
   - Configured for automatic scaling
   - Port 5000 internally mapped to port 80 externally

4. **Database Connection**
   - Uses environment variables for database configuration
   - Compatible with Neon Database for PostgreSQL

5. **Environment Configuration**
   - Development vs. production mode controlled by NODE_ENV
   - API keys and secrets managed through environment variables

The deployment workflow is defined in the .replit file, which specifies build and run commands for both development and production environments.
